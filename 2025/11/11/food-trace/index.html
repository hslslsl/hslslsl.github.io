基于Fabric的高端食品安全系统

项目介绍

构建一个食品溯源系统

核心目标：实现高端食品在生产、加工、物流、销售各个环节信息的上链与溯源。

具体任务：

(1) 搭建一个包含供应商、加工厂、物流商、零售商的多节点Fabric网络。

(2) 部署食品溯源链码，定义关键数据结构。

(3) 模拟各参与方依次调用链码，更新食品的状态。

(4) 实现一个面向消费者的溯源查询界面，通过扫描商品二维码，即可查看其完整的流转历史。

本项目改自原项目：https://github.com/1123yyr/chainTrace.git

环境与工具

  • 操作系统
    • Ubuntu 24.04.1 LTS 用来跑 Fabric 网络、Peer、Orderer、CA
    • Windows 用来做前端开发与日常联调
  • 区块链相关
    • Fabric 2.5.14、Fabric-CA 1.5.15
    • Go 1.22.x（链码开发），模块 fabric-chaincode-go/v2
  • 后端与 SDK
    • OpenJDK 11.0.28
    • Spring Boot 2.1.17.RELEASE（REST 服务）
    • Fabric Java SDK 1.4.6（链上读写）
  • 前端
    • Vue 2.5.2、Element-UI 2.5.4、Axios、ECharts
    • Webpack 3 + DevServer，端口 8087
    • 代理配置：/restful -> http://localhost:9999/restful、/my -> http://localhost:8081（见 oldIot/config/index.js:12–33）

链码解释

  • 设计目标
    • 单条商品记录用一个键保存，键规则如 trace_ ，确保定位简单、更新高效。
    • 相同商品的不同环节信息分次写入链码：生产、运输、销售都调用各自的更新函数；只覆盖传入的非空字段，避免把已有信息抹掉。
    • 查询时返回“规整过”的字段，避免大小写与“null”造成展示脏数据。
  • 字段选择
    • 生产段： gname 、 pname 、 paddress 、 ptime 、 pperson
    • 运输段： tname 、 tperson 、 tbeginaddress 、 tarriveaddress
    • 销售段： sname 、 sperson 、 saddress 、 stime
    • 这些字段正好对应前端“差异校验”里对比的集合，见 oldIot/src/views/suyuan.vue:432
  • 为什么这样写
    • 用一个结构承载三个环节，更新时只覆盖“有值”的字段：好处是链上只保留真实填写过的信息；坏处是需要前端对空值做规整，我们用统一的 canon() 来归并空值。
    • Key 里带前缀 trace_ ：方便做范围查询（ GetStateByRange(“trace_”, “trace_z”) ）拿所有食品记录。
    • 函数划分为 PutProduce/UpdateTransport/UpdateSale：符合业务流程分段，责权清晰，也便于管控权限（以后可按 Caller 的 MSPID 限制不同函数）。
      链码示例（完整示意，逐句解释写在注释中）
  • PutProduce/UpdateTransport/UpdateSale：把对应环节字段用 canon() 清洗后写入；如果记录已存在，先反序列化旧值再覆盖新字段，避免丢失其它环节数据。
  • QueryById：返回该商品的完整记录，用于前端“数据库 vs 链上”的对比展示。
  • QueryAll：拿所有记录，前端可用来做列表或统计。

核心代码

  • 链上校验与差异对比（数据库 vs 链上）功能
    • 功能概述：同时请求数据库与链上数据，做字段规整与聚合，然后逐字段对比，得出差异项列表，弹窗展示交易元信息和对比结果。
    • 代码位置： oldIot/src/views/suyuan.vue:423–465

  • 为什么这么写

    • 两路拉取：数据库保证查询速度与历史；链上保证不可篡改，两者对比更有说服力。
    • 逐字段聚合：链上写入通常是分批次的（先生产、再运输、最后销售），用“覆盖非空值”的方式把最新有效值集聚起来。
    • 交易元信息：为消费者提供“这是链上记录、这是交易时间”，增强可信感。

  • 溯源查询“数据库/区块链”数据源切换与查询逻辑

    • 功能概述：根据用户选择的数据源（数据库或区块链），使用不同接口（ restful/… 或 my/sdk/… ）拉取对应数据；当没有输入 traid 时，拉取“全部”，当有 traid 时，拉取“单条并合并链上版本”。
    • 代码位置： oldIot/src/views/suyuan.vue:360–420
  • 核心点
    • 统一了链上返回结构，避免前端处理多种字段名（ tid/txId/txid ， ttime/txTime 等）。
    • 当没有 traid 时，采用数据库的“全部”查询提升速度；当有 traid 时，走链上详细记录做可信展示。

背书与“OR模式”的坑

  • 典型错误：“Failed to collect enough endorsements” 或 ENDORSEMENT_POLICY_FAILURE
    • 原因：链码没有安装在所有需要背书的 Peer；或背书策略过严（AND），但只提供了一个组织的背书
    • 解法：
      • 安装链码到所有需要背书的 Peer（Org1、Org2）
      • 批准、提交时用 OR 模式：--signature-policy "OR('Org1MSP.peer','Org2MSP.peer')"
      • 调用时同时带上两个 --peerAddresses 与对应 --tlsRootCertFiles
  • 另一类错误：chaincode definition exists, but chaincode is not installed on peer
    • 解法：到该 Peer 上执行 peer lifecycle chaincode install trace.tar.gz

证书/TLS 的坑

  • 如果出现 TLS handshake 相关错误：
    • 确认 CORE_PEER_TLS_ENABLED=true
    • 确认 --cafile 指向 orderer TLS CA，--tlsRootCertFiles 指向各 Peer 的 tls/ca.crt
    • 看日志：docker logs peer0.org1.example.com、docker logs orderer.example.com

故障排查速查表

  • 背书失败 → 复查：已安装到所有 Peer、OR 模式、调用时带上两个 --peerAddresses
  • 包 ID 不匹配 → 重新 queryinstalled 拿最新 PACKAGE_ID 并重新 approve
  • 链码升级（改代码）→ 提高 --sequence（比如 2、3），重新 approve/commit
  • TLS 报错 → 校对 --cafile 与 --tlsRootCertFiles 路径
  • 前端白屏 → 控制台看报错；确保代理指向的服务已启动；对所有接口返回做空容错
  • 端口冲突 → 改 oldIot/config/index.js 中 devserver 端口或停掉占用端口进程

建议与后续完善

  • 给链码增加事件，前端订阅实时更新
  • 增加图谱视图（生产→运输→销售）更直观
  • 把“数据库 vs 链上”的差异提示做成可导出报告
  • 加入性能压测与批量写入脚本（模拟大规模上链）

收尾

这套系统本质是“把关键字段写链+把链上数据漂亮地展示出来”。背书策略定清楚（OR 还是 AND），证书路径别写错，前端别对空对象 .length，基本就一路顺畅了。遇到 “Failed to collect enough endorsements”，不要慌，先把两个 Peer 的地址和证书带齐，再看策略是不是 OR。前端跑不起来，十有八九是 Node 版本和老 webpack 的八字不合，切到 Node 14，代理配好，马上就能看见“高端食品安全系统”的首页了。
