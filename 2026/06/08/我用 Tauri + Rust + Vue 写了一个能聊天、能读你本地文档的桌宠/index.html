我用 Tauri + Rust + Vue 写了一个能聊天、能读你本地文档的桌宠

一篇尽量讲清楚每个细节的开发笔记

桌面宠物（俗称桌宠）是个老题材了。早年的 VPet、Shimeji、Pochi 让一只小可爱在桌面上跑来跑去，戳一下卖个萌。这两年大模型起来以后，我就一直想：要是这只小宠物能开口说话、能读我桌面上的文档、还能在我熬夜赶代码的时候劝我休息，那不就成”赛博团宠”了吗？

于是有了这个项目。技术栈一句话概括：Tauri 2.x 套壳 + Rust 后端做脏活 + Vue 3 做 UI + 本地 ONNX 模型做语义检索 + 第三方 LLM API 做对话。

下面我把整个项目从外到里拆开讲，重点放在那些”教科书不会教、踩了才知道”的细节上。

一、整体架构：为什么是 Tauri + Rust + Vue

桌宠这种应用有几个硬性要求：

  1. 窗口透明、无边框、能穿透到桌面——纯 HTML/Web 没办法做到
  2. 要监听全局键鼠输入（不是只监听自己窗口里的）——浏览器禁区
  3. CPU 占用得低，毕竟它要常驻后台——Electron 太胖
  4. 要能跨平台编译成单个 exe——又不想写 C++

满足以上四点的方案大概就两个：Electron 或 Tauri。我选 Tauri 的理由很直接：

  • 打包出来的安装包只有十几 MB（Electron 动辄上百 MB），桌宠这种”小工具”应该轻量
  • 后端是 Rust，写底层 Win32 调用比 Node + node-ffi 舒服得多
  • WebView 用系统自带的（Windows 上是 WebView2），不再绑一份 Chromium

整个架构是这样的：

核心通信就两种：

  • Command：Vue 调 invoke('xxx', args)，Rust 端 #[tauri::command] 接收，返回值用 Promise 拿
  • Event：Rust 端 app_handle.emit('xxx', payload)，Vue 用 listen('xxx', cb) 监听

后面所有功能都在这两根管子上跑。

三、状态机：宠物为什么”知道”该做什么

PRD 里定义了 10 个状态：APPEAR、IDLE、CLICKED、THINK、TALK、SLEEPING、TYPING、WORRIED、SWEATING、SHUTDOWN。这只小宠物什么时候眨眼、什么时候打盹、什么时候提醒你别熬夜，全靠状态机决策。

我没有用现成的 FSM 库，因为这个状态机很特殊：

  1. 优先级：高优先级状态可以打断低优先级（比如 LLM 在思考时，键盘活跃不该把它切去 TYPING）
  2. 瞬态：APPEAR/CLICKED/SHUTDOWN 是”播完就走”的，不参与正常评估
  3. 持续条件：CPU ≥ 90% 要持续 30 秒才算 SWEATING；TYPING 退出要按键停 10 秒
  4. 跨线程信号：LLM 模块在异步线程里写 is_thinking/is_talking，状态机线程要感知

实现上，状态机就一个纯函数 evaluate_state，每秒被驱动一次：

设计上有几个关键点：

优先级隐式编码在判断顺序里。我没有维护一张 priority_table: HashMap<State, u32>——两份真相迟早不一致。判断顺序就是优先级，写在哪个 if 在前面，它就高优先。

每个 tick 只产出一个状态。早期版本里我有 evaluate_state + should_exit 两个函数，结果同一秒里会先切到一个状态再立刻切走，前端就闪烁。改成单函数后，调用方只比较 next != current，干净利落。

持续型条件由调用方维护计数器。cpu_high_seconds 和 exit_pending_seconds 不是 evaluate_state 内部的状态——它是个纯函数。主循环每 tick 自己累加：

THINK / TALK 用 AtomicBool 跨线程通信。LLM 异步任务在 Tokio 运行时里跑，状态机在自己的 OS 线程里跑，两者唯一的通信渠道就是这两个原子布尔。LLM 收到首个 chunk 就把 is_thinking=false, is_talking=true，状态机下一秒自然切到 TALK。流结束时清两个标记，状态机重新评估，回到 IDLE。简单粗暴，没有锁。

桌宠的”智能”全靠这一节：它得知道你在不在敲键盘、电脑闲了多久、CPU 是不是在烧。

键盘频率监听

要知道你”最近 2 秒按了几下键”，必须装一个全局低级别键盘钩子 WH_KEYBOARD_LL。这个钩子要单开一个线程并跑消息循环，否则会被 Windows 自动卸载：

几个要点：

  • 只记时间戳，不记键值。安全考虑，也避免被杀软误报
  • 时间戳放进 VecDeque，查询时清掉 2 秒前的，剩下的长度就是最近 2 秒的事件数
  • 退出时要发 WM_QUIT 给这个线程，不然 GetMessageW 永远不返回，进程退不掉。我把线程 ID 存到 HOOK_THREAD_ID，退出时 PostThreadMessageW(tid, WM_QUIT, ...)

用 Windows 的性能计数器 PdhCollectQueryData。这部分代码在 monitor/system.rs，不展开了，思路是开一个独立线程每秒采样一次，结果写到原子变量里，主循环直接读。

五、逐帧动画：让一千张 PNG 跑起来

这只小宠物的动画素材直接复用了 VPet 的开源逐帧 PNG。每个状态对应一个文件夹，里面是一堆按命名规则的 PNG：

文件名约定：<前缀>_<帧序号>_<毫秒数>.png。前缀随便写（中文也行），引擎不解析；序号决定播放顺序；毫秒数是该帧停留时长——这点很关键，眨眼、点头这种关键帧要停得长一点，过渡帧停得短一点，动作才自然。统一 fps 反而会丢失节奏。

引擎实现：递归 setTimeout

为什么不用 requestAnimationFrame 或 setInterval？因为：

  • requestAnimationFrame 锁死 60fps，无法每帧不同时长
  • setInterval 不能动态改间隔
  • setTimeout 一次一调度，最灵活

切图就一行 imgEl.src = frame.url，浏览器自己处理换图。

预加载：把 PNG 全塞进内存（但不是 RGBA）

如果每次切帧才去读硬盘，肯定卡。所以启动时把所有帧全部预加载，转成 blob URL：

关键：blob 保留 PNG 压缩格式，不解码成 RGBA。一张 1000×1000 的图解码后是 ~4MB，全部预加载几百帧就 OOM 了。保留 PNG 格式后，浏览器渲染时按需解码，实测 15 个状态 × 平均 15 帧的素材占内存 50-100MB，可以接受。

Idle 子变体：让待机不无聊

宠物的 IDLE 状态默认是”呼吸微动”，但一直循环看着死板。我加了”子变体”机制：在 IDLE 状态下，每 20-60 秒随机触发一次变体动画（喵叫、吹泡泡、蹲下眨眼……），主动画占 70%、变体占 30%：

变体切换不经过 Rust 状态机——它本质上还是 IDLE，只是换张图。

八、RAG：让宠物读懂你本地的几百份文档

为什么是本地 ONNX，不是云 API

  • 隐私：你的笔记、合同、内部文档不出本地
  • 成本：embedding 是高频操作，调云服务太贵
  • 速度：本地推理 < 100ms

我选了 bge-small-zh 这个中文友好的 embedding 模型，导出成 ONNX 格式后约 91MB，跑 CPU 都很快。

Rust 调 ONNX：用 ort crate

ort 是 ONNX Runtime 的 Rust 绑定。看起来很简单：

但实际上这一节我踩了最久的坑。

坑 1：ONNX Runtime DLL 直接挂起

Session::builder()?.commit_from_file() 这一行调用后，整个进程永远不返回。换线程调没用、换路径没用、加超时也没用——整个进程像被冻住了。

排查了 4 个小时才发现：models/ 目录里那份 onnxruntime.dll 是个有问题的版本。它在 DllMain 或 TLS 回调初始化时会挂起（GitHub Issue 有人遇到过同样问题）。

解决方案：在 build.rs 里自动从微软 GitHub Releases 下载官方的 1.24.2 版本，匹配 ort v2.0.0-rc.12：

另一个坑：ort crate 有 download-binaries feature 可以”自动下载 DLL”，但它和 load-dynamic feature 互斥——后者会设置 disable-linking，构建脚本第一行就 return，跳过下载逻辑。所以只能自己写。

坑 2：Session 不是 Sync

ort v2 的 Session 实现了 Send 但不是 Sync（ONNX Runtime C API 内部有线程局部状态）。也就是说，多个线程不能同时调用同一个 session 的 run()。

九、退出进程：一个让我崩溃的小问题

桌宠日常用着没事，但关闭它这件事，让我调试了 3 个小时。

现象：用户点托盘的”退出”，命令行卡住、进程不结束、Ctrl+C 也不响应、留下僵尸进程。

排查路径：

  1. 后台线程没退出 → 加 shutdown_flag 信号 → 线程是退出了，但进程还活着
  2. app.exit(0) → 进程不退
  3. std::process::exit(0) → 进程不退
  4. 直接 Win32 ExitProcess(0) → 进程还是不退！
  5. 到这里我已经怀疑人生了。Process Explorer 看进程树——发现是 WebView2 的子进程在做 DLL_PROCESS_DETACH 清理时死锁

ExitProcess 会触发所有 DLL 的 DllMain(DLL_PROCESS_DETACH)，WebView2 的清理代码在这一步会卡住，于是 ExitProcess 永远不返回。

解决方案：用 TerminateProcess 替代 ExitProcess：

TerminateProcess 跳过所有 DLL 清理回调，直接砍——粗暴但有效。

所有退出路径（托盘”退出”、右键菜单”退出”、CloseRequested、shutdown 动画播完）都统一调这个函数。

教训：对于 WebView2 应用，ExitProcess 不可靠。GUI 应用的退出逻辑要永远准备 TerminateProcess 这把”大砍刀”作为最后兜底。

总结：这套技术栈的体感

写完这个项目，对 Tauri + Rust + Vue 的组合有几点感受：

Tauri 真的轻。最后打出来的安装包 ~12MB，运行时常驻内存 ~80MB。Electron 写同样的功能至少 5 倍。

Rust 写底层很爽。windows crate 把 Win32 API 包装得很整洁，配合 Result<T, String> 错误处理，写键鼠钩子、性能计数器、进程管理这种活比 C++ 顺手多了。

Vue 3 + Pinia 做 UI 完全够用。这个项目其实没有什么复杂状态——大部分状态都在 Rust 端，前端只是”显示器”。Composables（useAnimation/useChat/usePetDrag）把逻辑切得很干净。

最大的坑都在生态边缘：ONNX Runtime DLL 版本不匹配、ort crate feature 互斥、WebView2 退出死锁——这些不是技术栈本身的问题，但你必须自己踩过才知道。所有踩坑细节我都记在 RAG_TROUBLESHOOTING.md 里了，权当备忘。

如果你也想做一个有 AI 能力的桌面小工具，这套技术栈值得一试。代码大概 3000 行 Rust + 2000 行 Vue，一个人在业余时间就能完成。剩下的都是产品和创意了。

项目代码、PRD、技术设计文档都在仓库里，有兴趣可以直接读源码。
