从画原型到聊原型——产品经理的下一个基本功是 vibe coding

从画原型到聊原型——产品经理的下一个基本功是 vibe coding

这两年讨论原型工具的文章很多，绕来绕去都在回答同一个问题：Axure 还是墨刀？新手用什么、进阶用什么、复杂交互怎么做、协作怎么搞……

但这个问题本身可能已经过期了。

真正在变的不是”哪个原型工具更好”，而是画原型这件事的方式正在被 vibe coding 重写——不画了，改成”聊”出来。

先说一个事实：业界已经在转向了

如果你最近还在纠结新人应该先学 Axure 还是先学墨刀，可能要先看看一些更新的趋势：

  • 2025 年初，OpenAI 联合创始人 Andrej Karpathy 提出了 “vibe coding” 这个词，描述的是一种”用自然语言描述需求、让 AI 生成代码”的工作方式。一年之后，这个词已经成为产品圈的高频词。
  • Google、Stripe、Netflix 等公司的 PM 面试流程已经加入了 vibe coding 环节：给候选人一个产品问题、30-60 分钟时间、用 AI 工具搭出一个可交互原型，边搭边讲思路。
  • 一些产品团队（比如 BuildBetter）已经完全跳过了 Figma + Linear 的传统组合，直接用 AI 生成 HTML 原型来推进产品迭代。
  • Anthropic 推出了 Claude Design，定位就是”用一段话描述，生成可点击的高保真 HTML 原型”。

这不是某一两家公司的实验性玩法，而是行业层面的工作方式迁移。在我接触的产品同行里，已经有不少团队不再做 Axure 文件评审了——他们直接发 HTML 原型链接，评审者点开就能交互，比看静态图直观得多。

为什么是 HTML，不是 Axure 或墨刀

讨论这个问题之前，先想清楚原型到底是用来干什么的。

原型的核心目的有两个：

  1. 传达设计意图 —— 让评审方、开发方、业务方理解你想做什么
  2. 暴露设计问题 —— 让你自己在动手开发之前，发现交互逻辑、信息结构上的漏洞

传统工具（Axure、墨刀、Figma）能不能做这两件事？能。但代价是：

第一，学习成本高。 Axure 的高级交互（变量、动态面板、条件逻辑）需要专门学习，新人上手到能产出可用原型，往往需要数周。墨刀简单一些，但简单的代价是复杂场景做不出来。

第二，视觉效果到不了真实系统的水平。 原型工具的组件库再丰富，也是组件库。你画出来的东西总是带着”原型味”，跟真实系统的观感差距明显。评审时这种差距会让评审者一直分神在视觉上：”这个按钮真的会这样吗？””这个颜色是最终颜色吗？”

第三，重型工具的迭代慢。 一个稍微复杂一点的页面，从画到改到再改，时间成本很高。需求变了，原型跟着变，很容易出现”原型还没画完，需求又变了”的尴尬。

而 HTML + AI 的组合，正好把这三件事都解决了：

  • 学习成本低。你不需要懂 HTML/CSS/JS——你只需要会描述你要什么。技术细节交给 AI。
  • 视觉接近真实系统。HTML 渲染出来的就是浏览器里的样子。配合 Tailwind 这类成熟样式库，几分钟就能产出贴近真实系统的视觉效果。如果想更贴近，把目标系统的页面截图丢给 AI，让它仿照风格生成，效果惊人。
  • 迭代快。改一个按钮的位置、调一段文案、加一个交互状态——一句话的事。看着不满意，直接对话：”把右上角的按钮改成主色按钮，hover 时加阴影”——几秒钟就出来了。

更关键的是，现在的 AI 工具（Claude Code、Cursor、Lovable、v0 等）已经支持指针选取交互：你直接在原型页面上点中某个元素，然后告诉 AI”把这块改成这样”，AI 就会精准地修改对应的代码。这种”所见即所改”的体验，传统原型工具反而做不到。

不是工具升级，是产品经理的工作方式变了

讲到这里你可能会想：好啊，又一个新工具，学就是了。

但我想说的不止这个。

vibe coding 真正改变的，是产品经理”画原型”和”做产品”的边界。

过去的工作链路是这样的：

每一步都有信息损耗：PM 心里想的，画到原型上掉了一点；原型交给设计师，设计师按自己理解再做一遍，又掉了一点；设计师交给开发，开发按代码实现，又掉了一点。等到最后产品上线，跟最初的设想可能已经差出十万八千里。

vibe coding 把这条链路压缩了：

这里的关键变化在于：PM 第一次有能力直接产出”接近最终产品”的东西。不是示意图，不是色块，而是一个浏览器里就能交互的、视觉上跟真实系统几乎一致的界面。

这个变化的影响比”省了点画原型的时间”大得多：

  • 沟通损耗大幅降低。开发看的不是抽象的描述，而是一个能跑的页面，照着实现就行。设计师拿到的也不是空气，而是一个可以在此基础上继续打磨的起点。
  • 评审更聚焦在”对不对”，而不是”长什么样”。视觉接近最终效果，评审者就不会再纠结颜色和间距，能把注意力放在交互逻辑和业务规则上。
  • PM 对最终产品的把控力增强。你能直接把脑子里的想法落到能跑的代码上，意味着你不需要再依赖一个完美的”翻译链”才能让自己的设计被实现。

所以与其说”原型工具迭代了”，不如说”产品经理的能力边界扩大了”。vibe coding 不是一个新工具，是一种新的工作能力。

高效用 AI 做 HTML 原型的几个关键点

工具好不好用，最终还是看人怎么用。同样让 AI 做原型，有人 10 分钟出一个能用的版本，有人改两小时还不满意。差距在哪？基本都在 prompt 和 skill 的使用上。

下面是几个实用的关键点。

1. 给足上下文，不要让 AI 猜

最常见的低效写法是：

“做一个登录页”

AI 给你一个能用但没有任何记忆点的登录页——因为它只能猜。

更好的写法是：

“为一个内部财务系统做一个登录页面。系统整体风格是企业级 SaaS，主色 #2563EB，字体使用 Inter。页面布局：左侧是品牌区（深色背景 + Logo + 一句标语），右侧是登录表单（账号/密码/记住我/登录按钮）。表单要支持错误提示。技术栈用纯 HTML + Tailwind CSS。”

这两个 prompt 出来的效果差距可能是肉眼看到的两个等级。

一个被业界反复验证过的原则：上下文比简洁更重要。多写 10 秒钟的 prompt，可以省下 10 分钟的修改时间。

2. 提供视觉参考

如果你希望 AI 做出来的东西贴近某个具体风格，最快的方式是给视觉参考：

  • 把目标系统的页面截图丢进对话里，告诉 AI “请仿照这个风格”
  • 把竞品页面的 URL 给 AI（如果工具支持），让它解析样式
  • 把设计风格描述出来：”参考 Linear 的简约风格” / “类似 Notion 的克制风格” / “类似 Stripe 的精致企业风”

视觉这件事，”描述” 比 “讲道理” 有用得多。

3. 数据先行，UI 跟上

一个进阶技巧：在让 AI 设计 UI 之前，先让它生成一份合理的 JSON 数据。

为什么？因为高保真原型的真实感很大程度来自于”数据填得是真的像”。如果你的列表里全是”测试1、测试2、测试3”，无论 UI 多漂亮都显得假。先生成一份贴近真实业务的 JSON（比如 30 条订单数据，金额、状态、时间分布合理），再把这份数据交给 AI 渲染界面——出来的东西会更接近真实系统的体感。

这个方法对 B 端复杂表格类页面特别有效。

4. 沉淀自己的 skill

在团队里反复做同一类型的原型时，可以把好用的 prompt 沉淀成可复用的 skill。

比如你所在团队的产品有固定的视觉风格、固定的导航布局、固定的组件库——把这些规范写成一份”团队 UI 规范”的 prompt，每次做新原型时把它作为前置上下文喂进去。这样一来，所有产出的原型都自带团队风格，不需要每次重新描述。

我自己的做法是维护一个 prompts/ 文件夹，里面按场景分类：登录注册、列表表格、详情页、表单、Dashboard……每次做新原型时调用对应的模板，少走很多弯路。

5. 二次修改要具体

第一版生成出来基本不会完美。这时候不要笼统地说”再优化一下”——AI 不知道你想优化什么。

具体一点：

  • “把右上角的搜索框改成圆角，宽度增加到 320px”
  • “表头加上吸顶效果，滚动时保持可见”
  • “把”提交”按钮的状态分为：默认、hover、loading、disabled，每个状态给不同的视觉反馈”

具体的指令，对应具体的修改。模糊的指令，对应模糊的修改。

更好用的是配合 IDE 的指针选取功能：直接在预览界面点中要改的元素，AI 就知道你说的是哪一块。这种交互方式比文字描述高效得多。

6. 知道什么时候该停

vibe coding 有一个隐形的陷阱：做得太顺了，容易把原型当成成品继续打磨。

但原型的目的是验证设计意图，不是上线。当原型已经能让评审者理解你的设计、能让开发者照着实现、能让你自己发现交互问题时，它的使命就完成了——不需要继续磨细节。

留给开发者去做真正的工程实现，留给设计师去做最后一公里的视觉打磨，PM 把精力收回到产品决策本身。这才是 vibe coding 的合理使用边界。

不变的是什么

聊了这么多 vibe coding 的好处，最后想说一个反向的观点：

会用 AI 做漂亮原型，并不等于成为更好的产品经理。

产品经理的核心能力从来不是”画原型”，而是：

  • 理解业务、识别真问题
  • 抽象用户痛点、设计解决方案
  • 在多种约束下做出合理的取舍
  • 把复杂的事讲清楚、推动各方达成共识

这些能力在 AI 时代不仅没有贬值，反而更值钱了——因为当所有 PM 都能十分钟做出漂亮原型时，真正的差距会回到”你想做什么”和”为什么这么做”上。

vibe coding 是放大器。你产品思维的水平有多高，它放大出来的产出就有多高。如果思维本身是空的，那再漂亮的 HTML 原型也只是好看的空壳。

所以与其说原型工具迭代了，不如说工具帮 PM 摆脱了”花在画图上的时间”，让 PM 有更多余力专注在产品本身。这才是这场迁移真正的价值。

写在最后

如果你还在用 Axure 或墨刀做原型，没问题——它们是成熟的工具，能完成基本工作。但如果你想知道未来的产品经理工作方式是什么样，可以花一个周末试试 vibe coding：打开 Claude 或 Cursor，从一个简单的页面开始，用对话的方式做出第一个 HTML 原型。

你会发现，做原型这件事的逻辑变了：从”我要把脑子里的画面画出来”，变成”我要把脑子里的想法说清楚”。

工具只是工具，真正在迭代的，是产品经理的工作方式本身。
