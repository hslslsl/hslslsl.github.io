把 AI 的能力变成自己的：如何使用和沉淀 Skills

把 AI 的能力变成自己的：如何使用和沉淀 Skills

用 AI 久了你会发现一件事：每次开始一个新任务，仿佛都在重头训练 AI——告诉它你的偏好、你的项目背景、你的工作规范、你想要的输出风格……这些信息你说过无数遍，但每次新对话它都不知道。

Skills 就是为了解决这件事的。

这篇文章聊聊 Skills 是什么、怎么找到好用的 Skills 直接拿来用、以及更重要的一件事——怎么把自己的工作经验沉淀成属于自己的 Skills。

先说 Skills 是什么、从哪来

Anthropic 在 2025 年 10 月推出了 Skills（也叫 Agent Skills），本质上是一种可被 AI 按需加载的”能力包”。

每个 Skill 是一个文件夹，里面包含：

  • SKILL.md：核心说明文件，告诉 AI 这个 skill 是干什么的、什么时候用、怎么用
  • 可选的脚本、参考文档、模板素材等附带资源

它的工作机制叫渐进披露（Progressive Disclosure）——简单说就是按需加载：

  1. 第一层：元数据（skill 的名称 + 描述）始终在 AI 的上下文里。这部分很轻，几十到一百个词左右
  2. 第二层：SKILL.md 正文——只有当 AI 判断这个 skill 跟当前任务相关时才加载
  3. 第三层：附带资源（脚本、参考文件）——只有真正需要时才读取

这个设计很聪明。它让 AI 可以同时拥有几十上百个 skill，但任何时候上下文里只有真正用得上的那几个，不会因为能力变多而拖慢推理或挤占 token。

Skills 和 MCP、Prompt 有什么不同

刚接触 AI 的同学常会把这几个东西搞混，简单说一下：

  • Prompt 是你跟 AI 说的话，一次性的。下次新对话要重新说一遍。
  • MCP（Model Context Protocol） 是 AI 调用外部工具/数据源的协议，比如让 AI 读取数据库、调用 API。它解决的是”AI 能访问什么”的问题。
  • Skills 是 AI 完成某类任务的”操作手册”。它解决的是”AI 怎么做这件事做得好”的问题。

三者是互补的：MCP 让 AI 拿到原料，Skills 告诉 AI 怎么处理原料，Prompt 是你具体要它做什么。一个真正”懂你”的 AI 工作流，三者都不可少。

上半场：怎么找到好用的 Skills 直接用

如果你刚开始接触 Skills，最快的入门方式不是自己写，而是先用别人写好的——看看好的 Skill 长什么样，找几个真正解决你日常问题的，先体验一下”AI 自动调用 Skill 帮你干活”的感觉。

几个值得逛的 Skill 集散地

1. awesome-claude-skills（VoltAgent 维护）

GitHub 上规模最大的 Skill 索引之一，号称收录了 700+ 个来自官方团队和社区的 skill，覆盖软件开发、写作、数据分析、设计等各种场景。

2. awesome-claude-skills（Composio HQ 维护）

经过策展整理的版本，质量比较稳定。里面有一些做得很好的实战案例，比如 invoice-organizer（发票整理）、brand-guidelines（品牌规范）等，可以直接拿来读源码学习。

3. claude-market/marketplace

开源的、人工策展的 marketplace，专门收录优质的 skill、agent、tool。胜在质量筛选过，量不大但精品多。

4. Anthropic 官方 cookbook

官方出的示例和最佳实践合集。如果你想看”教科书级别”的 Skill 写法，这里是首选。

怎么挑 Skill

社区里 Skill 数量很多，但不是每个都值得装。挑的时候看几点：

  1. README/SKILL.md 写得清楚不清楚 —— 写不清楚的 skill，触发概率低、效果不稳定。一份好 skill 的描述应该一眼能让你看明白它什么时候该用、能做到什么
  2. 是否有具体例子 —— 没有 input/output 例子的 skill 通常质量一般
  3. 是否分层组织 —— 复杂功能的 skill 应该有 references/scripts 等子目录，而不是把所有东西堆在一个 SKILL.md 里
  4. 作者是不是认真维护 —— 看 commit 频率和 issue 响应。冷门作者的”一锤子 skill”用着用着就跟新版本不兼容了

怎么使用

具体方式取决于你用的客户端（Claude Desktop / Claude Code / Cursor / Kiro 等），但思路都差不多：

  1. 把 skill 文件夹放到对应的 skills 目录（比如 ~/.claude/skills/ 或工作区的 .skills/ 下）
  2. 重启或刷新客户端
  3. 对话时 AI 会自动判断哪个 skill 适合当前任务，自动加载

一个常被忽略的细节：skill 是被”动态触发”的，不是手动调用的。 所以你不需要每次说”请使用 XX skill”，而是把任务描述得足够具体，AI 会自己判断是否要调用。如果你发现某个 skill 老是不被触发，问题往往出在它的 description 写得不够吸引人——后面写自己的 skill 时会聊到这点。

下半场：如何沉淀自己的 Skills（重点）

用别人的 skill 是”消费”，写自己的 skill 才是”积累”。

我个人觉得 Skills 这套机制最大的价值不在于装多少个开源 skill，而在于——它给了你一个把”个人工作方法”固化下来的容器。

为什么这件事重要？因为我们每个人在自己的工作领域里，都有大量重复的、有规则的、但又散落在各处的”手感”。这些手感如果只活在你脑子里，每次新任务都要重头跟 AI 描述一遍；但如果能沉淀成 skill，就变成了你和 AI 之间的”共享词汇表”，效率会天差地别。

什么样的工作适合沉淀成 Skill

不是所有事情都值得做成 skill。判断标准很简单：这件事我做过两次以上，每次都得跟 AI 把规则讲一遍吗？

如果答案是肯定的，那就值得沉淀。常见的可沉淀场景：

  • 写作类规范：你公司的 PRD 模板、周报格式、会议纪要结构
  • 代码规范：项目的命名约定、注释风格、技术栈偏好
  • 流程类工作：调研问卷的设计模板、复盘文档的结构、用户访谈的提纲框架
  • 特定领域知识：业务术语表、内部系统的字段含义、合规规范
  • 重复性产出：周报自动生成、需求总结归类、问题抽象

举个我自己的例子：每次做用户调研，我都会先列一份调研问题清单，按”基础信息→使用场景→行为流程→痛点期望”四层来设计问题。这个模板在我脑子里很清楚，但每次让 AI 帮我做调研提纲，我都得重新讲一遍。这就是一个典型的可沉淀场景——把它写成一个 user-interview-planner 的 skill，以后只要说”帮我准备一份给财务实习生的调研提纲”，AI 就会按照我的方法论来做。

写一个好 Skill 的几个关键

写过几个 skill 之后，会发现不同 skill 的质量差距非常大。同样想做一件事，有的 skill AI 一看就懂、产出稳定；有的 skill 写了一大堆，AI 还是经常跑偏。差距在哪？大致是这几点。

1. description 是命脉

Skill 的 frontmatter 里有两个必填字段：name 和 description。其中 description 决定了 AI 在面对一个新任务时，会不会想到要用你的 skill。

写得不好的 description：

“Help with documents.”

写得好的 description（参考 skill-creator 的写法）：

“Create new skills, modify and improve existing skills, and measure skill performance. Use when users want to create a skill from scratch, edit, or optimize an existing skill, run evals to test a skill, benchmark skill performance with variance analysis, or optimize a skill’s description for better triggering accuracy.”

差距在哪？好的 description 做了两件事：

  • 说清楚 skill 能做什么（具体的能力清单）
  • 列出了什么场景下应该用这个 skill（触发的时机）

而且还有一个反直觉的点：description 应该写得稍微”推销感”一点。AI 有”欠触发”的倾向，宁可不用 skill，也不愿意冒”用错”的风险。所以在 description 里明确说”应该用这个 skill 当用户提到 X、Y、Z 的时候”，比含糊地说”可以帮助处理文档”要有效得多。

2. 把”为什么”写出来，而不只是”是什么”

差的 skill 写法是这样的：

读起来像在对一个不听话的下属下命令。问题是——AI 是知道揣摩意图的，硬性规则反而限制它的发挥。

更好的写法是解释为什么：

把规则背后的逻辑讲清楚，AI 不仅会遵守这个规则，还会在边界情况下做出合理判断。

skill-creator 的原文里有一句话讲得特别精准：

试着解释你让模型做每件事背后的”为什么”。今天的 LLM 是聪明的。它们有不错的心智理论，在好的引导下可以超越机械执行，真正把事情办成。

这个原则适用于所有 skill 写作。

3. 控制长度，分层组织

经验值是：SKILL.md 控制在 500 行以内。

为什么？因为 SKILL.md 一旦被触发，会全文加载到 AI 的上下文里。写太长会挤占 token 不说，AI 在长文档里也容易抓不住重点。

如果你的 skill 真的需要很多内容（比如要支持多个子场景），用分层组织：

SKILL.md 里只放核心逻辑和”什么情况下读哪个 reference”的指引，详细内容放在 references 里，AI 会在需要时再去读。这样既保留了完整能力，又不会一开始就把所有信息糊在 AI 脸上。

4. 加例子，不要光讲规则

任何抽象的规则都不如一个具体的例子直观。在 skill 里多放 input/output 的对照例子：

AI 看一眼例子，比读三段规则更快理解你想要什么。

5. 写完之后用 fresh eyes 看一遍

写 skill 跟写代码一样，第一稿基本不是最优解。建议的做法：

  1. 写完第一版后，让自己脑子放空一会儿（喝杯咖啡，看看别的）
  2. 用”第一次看到这个 skill 的人”的视角重读一遍
  3. 找两三个真实的任务来测试它，看 AI 的产出符不符合预期
  4. 不符合的地方，问自己”是 skill 写得不够清楚，还是 AI 真的没看懂”——通常是前者

如果有条件，可以参考 skill-creator 那一套评测循环：写测试用例 → 跑一遍 → 看结果 → 改 skill → 再跑——这套流程能帮你快速定位 skill 的问题。

一个具体的例子：从需求到 Skill

把上面这些原则串起来，举一个完整的例子。

场景：作为产品经理，每周都要写工作周报。同事看的部分要写得详细，老板看的部分要 high-level。每次都得跟 AI 重新讲格式。

沉淀方案：写一个 weekly-report-generator 的 skill。

SKILL.md 关键部分：

这个 skill 写好之后，下次你说”帮我写本周的工作周报，我做的事情包括 A、B、C”，AI 自动用这套结构生成两个版本——你不再需要每次重新解释格式。

把 skill 当作”个人知识资产”经营

最后一个观点：写 skill 不只是为了眼前这次任务用着方便，而是在为自己积累一个长期资产。

每写一个 skill，等于把”我是怎么做这件事的”显式地写下来了。这件事的好处不止是 AI 能用——更重要的是：

  • 它逼你把工作方法显式化。很多事情你”会做”但说不出来，写 skill 的过程就是把隐性知识转成显性知识的过程。这本身对你的成长就是一次梳理。
  • 它形成了可分享的资产。你写的 skill 可以分享给团队、分享给社区，让更多人复用你的方法论。这是 PRD/SOP 文档做不到的——文档是给人看的，skill 是给 AI 用的，AI 一调用就直接产出，复用门槛极低。
  • 它会随你成长。每次你的方法升级，更新一次 skill 即可。三年后你回头看，会有一整套属于你自己的 AI 工作方法库。

我建议每个想认真用 AI 的人，都建一个属于自己的 ~/.skills/personal/ 目录，把日常会用到的工作流一个个沉淀进去。最初可能就两三个，但累积一年下来，那就是几十个属于你自己的、AI 一调用就能精准产出的能力。

这个时候你会发现，你和 AI 的工作关系真的发生了变化——它不再是一个每次都要重新解释的助手，而是一个真正”懂你”的协作者。

写在最后

讲了这么多技巧，最后回到一个朴素的判断：

Skills 的价值不在于”AI 多了一个新功能”，而在于它给了你一个把工作经验、行业知识、个人偏好结构化沉淀的方式。

会用别人的 skill，可以让你站在巨人肩膀上，少走弯路；
会写自己的 skill，可以让你的经验脱离一次性的对话，变成可复用、可传播、可迭代的资产。

下一次，当你又一次跟 AI 解释某件事的规则时——停下来想一下：”这件事我已经讲过几遍了？是不是该写成 skill 了？”

把那一刻的”重复性烦躁感”，转化为一次沉淀的契机，时间久了，差距就拉开了。
